17 Services, One File: 5 Counter-Intuitive Lessons from Building a Fast SaaS Toolkit

Created on 2026-04-07 03:58

Published on 2026-04-07 04:01

1. Introduction: The SaaS Boilerplate Fatigue

Modern developers face a crushing paradox: the ecosystem is richer than ever, yet "SaaS boilerplate fatigue" has reached an all-time high. Founders routinely waste weeks wrestling with the "configuration nightmare" of wiring together authentication, billing, logging, and workflow automation before a single line of business logic is written.

The Autonomyx project began as a quest to dismantle this complexity. By shifting from a rigid, proprietary framework to an "OSS Assembly" model—bundling 17 essential services like Logto, Lago, and n8n into a unified launchpad—we proved that enterprise-ready infrastructure can be launched in minutes. But getting there required more than just a docker-compose.yml file; it required an architectural rethink focused on operational efficiency, using pnpm workspaces for strict dependency isolation to prevent "phantom dependency" bugs that plague standard npm monorepos.

2. Takeaway 1: Docker Profiles and the Caddy Advantage

When managing 17 services, the standard approach is to split configurations into a fragile stack of files—compose.base.yml, compose.ops.yml, and so on. This is a maintenance disaster. We opted instead for Docker Compose Profiles to keep every service in a single, authoritative file. Developers can selectively enable groups like --profile ops for monitoring or --profile growth for marketing tools, avoiding the "fragility of stacked overrides" where one -f flag out of order breaks the entire networking layer.

Equally critical was the choice of Caddy over Traefik. In a 17-service environment, Traefik’s label-based configuration—scattered across dozens of services—becomes unreadable. Caddy offers a single, human-readable Caddyfile and handles auto-provisioning HTTPS certificates with zero config. For a toolkit where users are constantly adding or removing services, regenerating one Caddyfile is infinitely simpler than managing a web of Docker labels.

Insight: Profiles avoid the configuration nightmare of managing complex merge orders. By using a single source of truth, the CLI can programmatically generate the correct environment variables, ensuring a clean, predictable local environment that scales without "phantom" service dependencies.

3. Takeaway 2: Why Multi-Tenancy Should Be Middleware, Not a Service

A common trap for architects is implementing a dedicated service for multi-tenancy. We argue for a "thin" approach: multi-tenancy as a TypeScript middleware layer. By using row-level isolation (a shared database with a tenant_id column), we provide 90% of what a SaaS needs at zero extra operational cost.

However, a "Senior Architect" knows that being "thin" doesn't mean being "rigid." A key lesson from the assembly was the Matomo Exception: Matomo requires MySQL/MariaDB, while the rest of our stack is PostgreSQL-native. A rigid framework would force a single database type on the developer; an "Assembly" gives Matomo its own lightweight MariaDB container. This is the essence of composability: use the right tool for the job, even if it breaks the "one database" rule.

"Premature isolation is the #1 over-engineering mistake in early SaaS."

4. Takeaway 3: The Redis Buffer—Protecting Your Database from Your Own Success

Writing a new row to PostgreSQL for every API call or billing event creates "write amplification" that kills performance at scale. To mitigate this, our toolkit implements a Redis-first usage tracking buffer.

Events are pushed to Redis using RPUSH—an O(1), non-blocking operation. We then periodically flush these in batches of 500 into PostgreSQL. To ensure external billing systems don't bottleneck the app, we utilize a "fire-and-forget" pattern for Lago; we forward billing events asynchronously because external billing APIs are notoriously high-latency.

For rate limiting, we bypassed "fixed-window" counters (which are vulnerable to bursts at the edge of the window) in favor of a sliding-window rate limiter using Redis sorted sets. By storing individual request timestamps as members and using ZREMRANGEBYSCORE to prune old entries, we create a true, gapless limit that prevents users from doubling their allotted requests during window resets.

5. Takeaway 4: The Rise of the "Meta-Skill"—Orchestrating AI

The toolkit’s integration with Claude Code introduced the "Meta-Skill"—a conductor that orchestrates specialized skills like oss-to-saas, deploy-to-coolify, and license-advisor. Rather than just writing code, this skill manages the environment through distinct modes:

  • Configure Mode: Turns hours of UI clicking in services like Logto or Lago into a single conversation.

  • Diagnose Mode: Specifically designed to troubleshoot service-level failures.

  • Convert Mode: Acts as an adapter, hijacking generic plans and rewriting them to use our pre-wired OSS assembly.

We also implemented an MCP (Model Context Protocol) fallback pattern. If an MCP server isn't connected for programmatic configuration, the AI gracefully degrades to providing manual, step-by-step instructions from the service’s README. This ensures the developer is never stranded by their own tooling.

6. Takeaway 5: Choose "Assembly" Over "Framework" to Avoid Lock-in

The final lesson is the shift from a proprietary framework to an "OSS Assembly." Each component—Logto, Lago, n8n—is a standalone project with its own roadmap. Our toolkit is merely the "glue."

Being an architect means acknowledging the "operational scars" of this approach. For example, during testing, we found that Logto requires a specific db seed command on first launch and originally conflicted with Uptime Kuma on port 3001. We had to move Logto to 3301 and bake the seeding into the setup.sh script. These are the real friction points of "Fast SaaS"—even with a single file, individual service quirks require manual handling. By keeping the layer thin, we allow developers to swap any piece without rewriting their entire application.

7. Conclusion: The Future is Composable

The Autonomyx Fast SaaS Toolkit demonstrates that launching enterprise-grade infrastructure isn't about building a bigger framework; it's about the intelligent assembly of world-class parts. By leveraging Docker profiles, Caddy’s simplicity, and AI-native orchestration, we can launch "ready-for-scale" stacks in minutes.

As you look at your next project, ask yourself: Are you wasting your time building a custom framework from scratch, or are you assembling the best of what already exists to get to market before the window closes?
